Frontend Development Plan (SRM Project) — Enhanced

## 🧠 Общая идея

Цель — построить frontend, который:

- корректно работает с backend (FastAPI)
    
- масштабируется без хаоса
    
- не ломается при изменениях API
    

Фокус:

- сначала работоспособность
    
- затем устойчивость (типы, структура, UX)
    

---

## 🛠 Технологический стек

- React + TypeScript
    
- Vite
    
- React Router
    
- TanStack Query
    
- Axios
    
- Zod
    
- Ant Design
    

---

## 📁 Структура проекта

```id="w8j2pl"
src/
 ├─ api/                # базовый клиент
 ├─ app/                # провайдеры (router, query)
 ├─ layout/
 ├─ features/
 │    ├─ auth/
 │    ├─ orders/
 │    └─ catalog/
 ├─ shared/
 └─ types/              # типы из OpenAPI
```

---

## 🚀 Этапы разработки

---

### 🔹 Этап 0 — Инициализация

- Создать проект (Vite + React + TS)
    
- Установить зависимости
    
- Настроить структуру
    

---

### 🔹 Этап 1 — База проекта

Сделать:

- Router (`/login`, `/orders`, `/catalog`)
    
- Layout
    
- Axios client (baseURL, headers)
    
- TanStack Query Provider
    

---

### 🔹 🔥 Этап 1.5 — Типы и API слой (ОЧЕНЬ ВАЖНО)

Сделать:

- Получить OpenAPI (`/openapi.json`)
    
- Сгенерировать или описать типы (DTO)
    
- Создать API слой:
    

```id="5k3t9d"
features/orders/api.ts
features/auth/api.ts
```

Цель:  
👉 единый источник правды для API  
👉 избежать рассинхрона данных

---

### 🔹 Этап 2 — Аутентификация

Сделать:

- Логин форма
    
- Сохранение JWT
    
- Auth store
    

Обязательно:

- обработка 401 → logout
    
- очистка токена
    

(Refresh можно добавить позже)

Результат:  
👉 безопасный вход и выход

---

### 🔹 Этап 3 — Orders (список)

Сделать:

- запрос списка заказов
    
- таблица (AntD)
    
- loading / error
    

---

### 🔹 🔥 Этап 3.5 — Матрица статусов и ролей

Сделать:

```id="z2b8l4"
status + role → доступные действия
```

Пример:

- NEW + RECEPTION → confirm
    
- CONFIRMED + PRINT → take_print
    

Цель:  
👉 не размазывать логику по кнопкам  
👉 централизовать правила

---

### 🔹 Этап 4 — Orders (детали)

Сделать:

- страница/модалка заказа
    
- отображение данных
    
- переход из списка
    

---

### 🔹 Этап 5 — Lifecycle

Сделать:

- кнопки действий по матрице
    
- вызывать API
    
- обновлять статус
    

Важно:  
👉 внедрять по шагам (2–3 действия за раз)

---

### 🔹 Этап 6 — Cancel flow

Сделать:

- request cancel
    
- approve / reject
    
- UI состояния
    

---

### 🔹 Этап 7 — Фильтры и поиск

Сделать:

- поиск (debounce 300–500ms)
    
- фильтры по статусу
    
- синхронизация с URL:
    

```id="s1d4kn"
?status=confirmed&search=ivan
```

Цель:  
👉 можно делиться ссылками  
👉 UX как в реальных системах

---

### 🔹 Этап 8 — Catalog CRUD

Сделать:

- список
    
- create / update / delete
    

Обязательно:

- Zod schema
    
- DTO mapper:
    

```id="mj2d8a"
Form → API DTO
API → UI
```

Цель:  
👉 не ломать данные при изменениях

---

### 🔹 Этап 9 — UI улучшения

Сделать:

- toast уведомления
    
- loading states
    
- empty states
    
- error handling
    

---

### 🔹 Этап 10 — Финализация + тесты

Сделать:

- рефакторинг
    
- убрать дублирование
    

Минимум E2E (3 сценария):

1. login
    
2. lifecycle (happy path)
    
3. cancel flow
    

---

## ⚡ Принципы

- сначала работает → потом красиво
    
- одна ответственность на компонент
    
- не дублировать API вызовы
    
- централизовать логику (roles/status)
